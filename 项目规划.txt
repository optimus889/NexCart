四、权限设计

Consumer

可以访问：

* 商品
* 购物车
* 支付
* 订单历史
* 个人信息

不可以访问：

* 商家商品管理
* 商家订单管理
* 管理员后台

Merchant

可以访问：

* 商品管理
* 商家订单
* 钱包
* 商家个人信息

不可以访问：

* 消费者支付页
* 管理员后台

Admin

可以访问：

* 平台总览
* 告警管理
* 用户管理
* 所有订单
* 所有商品

不可以访问：

* 消费者下单页
* 商家钱包操作页

五、验证码功能建议

登录界面可以做一个简单但可展示的验证码。

前端

* 登录页增加：
    * 验证码图片 / 文本
    * 输入框
    * 刷新验证码按钮

后端

* 新增接口：
    * GET /api/auth/captcha
    * 返回验证码 challenge
* 登录时校验验证码：
    * POST /api/auth/login

简化做法

你可以先做成：

* 后端生成 6 位随机字符串
* 存 session / memory / Redis
* 前端输入后提交
* 校验通过才允许登录

这样已经足够展示“登录安全控制”。

六、WebSocket 实时功能建议

1. 实时订单更新

场景：

* 消费者下单后
* 商家端订单页面实时收到新订单

2. 实时订单状态更新

场景：

* 商家把订单改成 Shipped
* 消费者 Orders 页实时看到状态变化

3. 实时告警

场景：

* 登录失败次数太多
* 库存不足
* 接口异常
* 管理员端实时收到 alert

十、建议开发顺序

为了能真正做出来，我建议按这个顺序，不要同时乱写。

Phase 1：数据持久化

* 接数据库
* 建 User / Product / Order / Alert 模型

Phase 2：认证与安全

* JWT
* bcrypt
* 验证码
* 登录失败限制
* RBAC 中间件

Phase 3：核心电商

* 购物车
* 下单
* 订单历史
* 商家订单管理
* 商品增删改查

Phase 4：管理员后台

* dashboard 统计
* 用户管理
* alerts
* logs

Phase 5：实时功能

* socket.io
* 新订单实时推送
* 订单状态实时更新
* 实时告警
